# 에러/로딩/빈 화면 처리

작성 기준: 2026-06-29

## 공통 API 오류

공통 API 오류 처리는 `src/services/http/apiClient.ts`에서 담당한다.

- 응답의 `detail` 또는 `message`가 있으면 해당 문구를 사용한다.
- 문자열 응답이면 해당 문자열을 사용한다.
- 그 외에는 HTTP status 또는 기본 오류 메시지를 사용한다.

401 응답은 refresh 요청을 한 번 시도한다. refresh도 실패하면 인증 상태를 정리하고 로그인 화면으로 이동한다.

403 응답은 권한 없음으로 보고 인증 상태를 정리한다. 이 경우 사용자는 다시 로그인해야 한다.

## Suspense 로딩

`App.tsx`는 web/mobile 로그인 화면과 작업장 화면을 lazy import로 로드한다.

현재 Suspense fallback은 `null`이다. 초기 로딩 중 별도 스피너를 표시하지 않는다.

알림 채팅방 모달은 lazy import 중 전용 fallback UI를 표시한다.

## 위험 로그 로딩

위험 로그 화면은 목록을 조회하는 동안 로딩 상태를 표시한다.

로딩 중 사용자는 아직 목록을 조작할 수 없으며, 조회가 끝난 뒤 상태 필터와 정렬이 적용된 결과를 본다.

## 위험 로그 에러

위험 로그 조회 또는 제출 실패 시 화면에 오류 메시지를 표시한다.

- 목록 조회 실패: 목록 영역에 오류를 표시한다.
- 조치 시작 실패: 모달 또는 시트의 입력 상태를 유지하고 오류를 표시한다.
- 조치 완료 실패: 입력 상태를 유지하고 다시 제출할 수 있게 한다.

403이 발생하면 공통 클라이언트가 인증 상태를 정리하므로 로그 페이지에 머무르지 않고 로그인 흐름으로 이동할 수 있다.

## 위험 로그 빈 화면

서버에서 내려온 로그가 없거나 현재 필터에 맞는 로그가 없으면 빈 상태를 표시한다.

대표적인 경우는 다음과 같다.

- 아직 위험 로그가 없음
- 기본 필터에서 `조치 후`만 존재함
- 상태 필터를 좁혀 결과가 없음

이때 사용자는 상태 필터를 조정하거나 새로고침으로 서버 목록을 다시 가져올 수 있다.

## 실시간 연결 문제

WebSocket 연결이 끊겨도 위험 로그 화면은 API 새로고침으로 복구할 수 있다. 실시간 이벤트는 사용자 조작 안정성을 위해 즉시 목록에 반영하지 않으므로, 최종 목록은 `GET /api/hazards` 응답을 기준으로 맞춘다.

## 시뮬레이션 빈 상태

시뮬레이션 화면에서 선택된 시나리오나 저장된 편집 레이아웃이 없으면 기본 레이아웃 또는 안내 상태를 사용한다. 엔진이 실행 중이 아닐 때는 마지막 스냅샷 또는 대기 상태를 표시한다.

## 편집기 빈 상태

편집기 화면은 새 레이아웃에서 시작할 수 있다. 저장된 레이아웃이 없으면 기본 상태로 객체 배치를 시작한다.
