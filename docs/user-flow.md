# 사용자 흐름

작성 기준: 2026-06-29

## 로그인 흐름

1. 사용자가 `/login`에 접근한다.
2. 아이디와 비밀번호를 입력한다.
3. 로그인 요청이 성공하면 access token과 세션 정보가 `sessionStorage`에 저장된다.
4. 앱은 `/simulation`으로 이동한다.
5. 인증이 필요한 API 요청에는 `Authorization: Bearer {accessToken}` 헤더가 붙는다.

## 화면 전환 흐름

상단 작업장 버튼으로 `시뮬레이션`, `편집 모드`, `위험 로그`를 전환한다.

- `시뮬레이션`: `/simulation`
- `편집 모드`: `/editor`
- `위험 로그`: `/logs`
- `시뮬레이션 전체화면`: `/simulation/fullscreen`

`/editor`와 `/simulation/fullscreen`은 진입 시 브라우저 전체화면을 요청한다. 사용자가 전체화면을 해제하면 시뮬레이션 전체화면 라우트는 `/simulation`으로 되돌아간다.

## 편집 후 시뮬레이션 흐름

1. 사용자가 편집 모드에서 배수 객체와 연결 정보를 구성한다.
2. 레이아웃은 로컬 저장소에 저장된다.
3. 사용자가 시뮬레이션 화면으로 이동한다.
4. 시뮬레이션 화면은 저장된 편집 레이아웃 또는 선택한 시나리오를 읽는다.
5. 사용자가 엔진 시작을 누르면 SWMM 엔진에 레이아웃과 제어값을 전달한다.
6. 실행 중에는 WebSocket 스냅샷을 받아 화면을 갱신한다.

## 위험 로그 확인 흐름

1. 사용자가 위험 로그 화면에 진입한다.
2. 화면은 `GET /api/hazards`로 로그 목록을 가져온다.
3. 기본 필터는 `조치 전`, `조치 중`이다.
4. 사용자는 상태 필터 버튼으로 `조치 후`까지 포함해 조회할 수 있다.
5. 사용자는 시간 헤더 버튼으로 최신순과 과거순을 전환할 수 있다.

## 위험 로그 조치 흐름

1. `조치 전` 로그를 클릭한다.
2. 모달 또는 하단 시트에 조치 내용을 입력한다.
3. 확인을 누르면 `POST /api/hazards/{id}/actions`를 호출한다.
4. 성공 시 해당 row는 `조치 중`으로 갱신되고, 내용에는 입력한 조치 내용이 표시된다.
5. 이때 다음 단계 모달은 자동으로 열리지 않는다.
6. 사용자가 `조치 중` row를 다시 클릭한다.
7. 결과와 선택 입력값인 재발 시 참고사항을 입력한다.
8. 확인을 누르면 `PATCH /api/hazards/{id}/actions/{action_id}`를 호출한다.
9. 성공 시 해당 row는 `조치 후`로 갱신된다.
10. 기본 필터에서는 `조치 후`가 보이지 않으므로 목록에서 사라질 수 있다.

## 새 로그 반영 흐름

1. WebSocket으로 위험 이벤트가 들어온다.
2. 화면은 바로 row를 추가하지 않고 새 로그 개수만 증가시킨다.
3. 사용자가 새로고침 버튼을 누른다.
4. 화면은 `GET /api/hazards`를 다시 호출한다.
5. 서버에서 내려온 전체 로그를 기준으로 목록을 다시 렌더링한다.
